### New features

- The user-guide configuration now checks external links with Ook's link-check service. The new `documenteer.ext.linkcheckservice` extension registers a builder over Sphinx's built-in `linkcheck` name, so existing `make linkcheck` invocations transparently submit the project's external links (collected from the resolved doctrees, with `linkcheck_ignore` patterns applied client-side) to the service and poll for its cached, retry-buffered results. The submission authenticates with a bearer token from the `OOK_TOKEN` environment variable, identifies the website by its origin base URL derived from `project.base_url`, and flags builds of the origin's default version from the GitHub Actions environment (overridable with `DOCUMENTEER_LINKCHECK_DEFAULT_BRANCH`). A submission whose URLs all resolve immediately is reported straight from the submission response; otherwise the builder polls the service until the check completes. New `[sphinx.linkcheck]` settings in `documenteer.toml` configure the integration: `use_service`, `service_url`, `poll_budget`, `strict`, and `origin_base_url`. The builder reports summary counts by status and lists each link needing attention with its page, HTTP status, and redirect location; `broken` links fail the build with a nonzero exit status while `redirected`, `failing` (retries pending), and `unsupported` links produce warnings only. A machine-readable `linkcheck.json` artifact with the full per-URL results is written to the build output directory. Service problems degrade gracefully by default: when the service is unreachable, the `OOK_TOKEN` is missing (as in a fork's pull request build), or the polling budget is exhausted, the builder warns that the link check was skipped and the build exits 0. Setting `[sphinx.linkcheck] strict = true` makes those same conditions fail the build instead, and `[sphinx.linkcheck] use_service = false` restores Sphinx's built-in in-process `linkcheck` builder.
